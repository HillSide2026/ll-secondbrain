# PRO-024: Monthly SLA Review Protocol

## 1. Purpose

Govern the monthly ML1 review of all SLAs that are not fully automatable.

This protocol does not replace automated enforcement. It closes the loop on the SLAs and KPIs that require judgment, observation, or manual data collection to assess.

---

## 2. Trigger

Monthly. Target: first working day of each calendar month.

The COS brief (generated by LLM-001) should include a standing agenda item: **Monthly SLA Review due** when review has not been completed for the current month.

---

## 3. Automated Pre-Work

Before the ML1 review session, run the following scripts and confirm outputs are in `06_RUNS/ops/sla_audit/`:

| Script | Covers | Command |
|--------|--------|---------|
| `scripts/ml2_schema_audit.py` | SLA-ML2-001/002/005, KPI-ML2-001/002/004 | `python scripts/ml2_schema_audit.py` |
| `scripts/integration_health_check.py` | SLA-SYS-001 | `python scripts/integration_health_check.py` |
| `scripts/run_log_validator.py` | PRO-023 compliance, KPI-ML2-004 | `python scripts/run_log_validator.py` |

These produce the factual inputs for §4. Do not proceed with the review without current outputs (max age: 48 hours before review).

---

## 4. Review Checklist

### 4.1 ML2 Record Layer (SLA-ML2 / KPI-ML2)

Read the most recent `audit_{run_id}.md` output.

| Check | SLA/KPI | Pass Condition | Action if Fail |
|-------|---------|----------------|----------------|
| Doctrine approval coverage | KPI-ML2-001 | ≥ 95% approved | Identify and promote remaining drafts |
| Draft backlog | KPI-ML2-002 | ≤ 10 drafts, none > 30 days old | Approve, supersede, or delete aged drafts |
| Schema compliance | KPI-ML2-004 | 100% | Fix violations, re-run audit |
| Ambiguous status values | SLA-ML2-005 | 0 | Correct inline |
| Conflict rate | KPI-ML2-003 | 0 | Identify conflicting doctrine, resolve |

### 4.2 System Execution Layer (SLA-SYS / KPI-SYS)

Read the most recent `integration_health_{run_id}.md` output plus any run logs from the past 30 days.

| Check | SLA/KPI | Pass Condition | Action if Fail |
|-------|---------|----------------|----------------|
| Integration failures | KPI-SYS-001 | ≤ 2% | Identify broken integrations, fix |
| Non-deterministic executions | KPI-SYS-002 | 0% | Review agent outputs for unexpected variance |
| Data integrity errors | KPI-SYS-003 | 0 | Audit run logs for cross-matter data |
| Boundary breach incidents | KPI-SYS-004 | 0 | Escalate immediately if any found |
| Silent failures | KPI-SYS-005 | 0 | Review for runs with no log entry |
| Recovery success rate | KPI-SYS-006 | ≥ 95% | Review failed runs for unrecovered failures |

### 4.3 Non-Automatable SLAs (Manual Assessment Required)

These require ML1 judgment and cannot be checked by script:

| SLA | What to Assess | Evidence Source |
|-----|----------------|-----------------|
| SLA-ML2-004 Retrieval Reliability | Were ML2 artifacts found when needed in the past month? | Recall any cases where a policy, playbook, or template was not surfaced or was found only after manual search. |
| SLA-SYS-002 Execution Determinism | Did any agent produce outputs inconsistent with its doctrine? | Review agent run logs for unexpected content, missed constraints, or doctrine deviations. |
| SLA-SYS-006 Escalation Fidelity | Were ambiguities escalated rather than resolved by the System? | Check run logs for any agent that appeared to make a judgment call without flagging it. |
| SLA-SB-002 Authority Discipline | Did ML1 decisions enter ML2 within the escalation resolution window (2 days)? | Compare any escalations from the past month against codified doctrine. |
| KPI-002-E Escalation Resolution Rate | % of escalations codified within 2-day window. | Check for open escalation items in ML1 decision queue. |

### 4.4 Matter Coverage (KPI-003)

| Check | Pass Condition | Action if Fail |
|-------|----------------|----------------|
| Essential matters: ≥1 artifact in last 1 day | ≥ 90% coverage | Identify uncovered matters, diagnose |
| Strategic matters: ≥1 artifact in last 2 days | ≥ 90% coverage | As above |

---

## 5. Output

At the end of the review, ML1 records:

1. **Review log entry** in `06_RUNS/ops/sla_audit/monthly_review_{YYYY-MM}.md` with:
   - Date completed
   - Pass/fail per check
   - Open violations with owner and resolution target date
   - Any doctrine gaps identified

2. **Decision queue updates** — any escalations, doctrine gaps, or violations that require ML1 decisions are added to `04_INITIATIVES/LL_PORTFOLIO/CHIEF_OF_STAFF/ML1_DECISION_QUEUE.md`

---

## 6. Escalation

If any of the following are found, escalation is **immediate** (not deferred to next review):

- KPI-SYS-004 Boundary breach (any matter isolation violation)
- KPI-002 Unauthorized output (any non-zero value)
- SLA-ML2-001 validator reporting critical failures
- Integration health check showing broken connections in use

Immediate escalation = ML1 addresses before end of same business day.

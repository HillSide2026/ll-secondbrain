# Scope and Positioning Note

Project: `HBP-013` - Sale of Granville

## Core Scope

The project scope is:

1. get `17409052 Canada Inc` licensed
2. make the licensed entity market-ready
3. eventually sell the licensed entity

The project scope is **not**:

- build and sell BaaS services
- create a separate market-facing banking product thesis
- treat vendor software as the asset being sold

## What Vendor Software Is For

Vendor software, EMI connectors, sandboxes, portals, or demo environments are
relevant only if they help one or more of the following:

- licensing credibility
- RPAA / regulatory credibility
- market-ready posture
- buyer diligence on the licensed entity
- transferability of the licensed entity

If a vendor proposal mainly sells "core banking" or broad BaaS functionality,
the project should ask whether a thinner package exists that supports the
entity-sale thesis without turning into a new operating business.

## Practical Reading of "Orchestration"

"Orchestration layer" is useful only as internal shorthand.

It should not become a fake product category.

In practical vendor terms, the relevant question is:

- is there a real, market-sold, lower-cost package that gives 174 enough UI,
  connector visibility, demo or sandbox access, and support for the licensed
  entity story without requiring a full core-banking build?

## Current Vendor Implication

For FinLego and Crassula, the right follow-up is not:

- quote us more core banking

It is:

- quote us the minimum thinner package that supports a licensed-entity sale
  thesis

That thinner package may include:

- limited UI or portal
- fiat-account representation if genuinely useful
- EMI connector visibility if real and transferable
- sandbox or demo access for diligence or regulator review

It should exclude unnecessary platform scope unless there is a clear licensed-
entity value reason to keep it.

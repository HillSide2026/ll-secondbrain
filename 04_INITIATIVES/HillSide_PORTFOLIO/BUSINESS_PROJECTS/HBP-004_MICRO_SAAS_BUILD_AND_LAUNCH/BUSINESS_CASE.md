# Business Case

Project: HBP-004 - Develop and Launch Micro SaaS (TariffLookup.ca)

## Strategic Rationale

HBP-004 tests whether HillSide can build and launch a narrowly scoped Micro
SaaS product without turning the work into an open-ended software venture.

The business case is not that TariffLookup.ca is already proven. The case is
that a constrained build-and-launch project can validate whether a focused
tariff / trade workflow has enough practical demand to justify deeper product
investment.

## Thesis

TariffLookup.ca may be valuable if it gives users a faster, clearer way to
answer a bounded tariff or trade-classification question than generic search,
manual lookup, or ad hoc professional-service intake.

The project should proceed only while the scope remains small enough to:

- freeze one core workflow;
- build an MVP quickly;
- launch to real users;
- observe activation evidence; and
- stop if demand is weak.

## Expected Value

| Value Driver | Description |
| --- | --- |
| Product validation | Tests whether a tariff-focused Micro SaaS wedge can attract real use |
| Portfolio learning | Builds HillSide experience with small software launch discipline |
| Trade services adjacency | May create insight or demand signals adjacent to trade remedies and market access work |
| Reusable operating pattern | If successful, creates a repeatable small-product build / launch pattern |

## Cost and Constraint View

| Constraint | Treatment |
| --- | --- |
| Cash cost | Keep minimal until product use is validated |
| ML1 time | Limit to direction, scope approval, and launch decisions |
| Build scope | Freeze MVP scope before meaningful execution |
| Distribution | Define the first user-acquisition path before production launch |
| Product risk | Do not expand into a broad trade platform before initial validation |

## Go / No-Go Logic

| Decision Point | Go Signal | No-Go Signal |
| --- | --- | --- |
| MVP scope | One core workflow can be described and built | Product requires multiple workflows before it is useful |
| Launch | Product can be put in front of real users | Launch depends on large content, integration, or advisory buildout |
| Validation | Users activate, return, or provide concrete workflow feedback | Users do not understand the offer or do not use the product |
| Next investment | Usage evidence supports further product work | Evidence supports parking or closing the project |

## Recommendation

Proceed only as a constrained strategic experiment. HBP-004 should stay focused
on building, launching, and validating a small Micro SaaS product rather than
authorizing a broad software business.

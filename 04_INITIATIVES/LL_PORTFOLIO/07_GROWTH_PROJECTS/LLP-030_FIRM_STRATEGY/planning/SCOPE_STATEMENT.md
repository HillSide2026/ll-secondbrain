# Scope Statement

Project ID: `LLP-030`
Project Path: `04_INITIATIVES/LL_PORTFOLIO/07_GROWTH_PROJECTS/LLP-030_FIRM_STRATEGY`
Stage: `Planning`

## Planning Use

Use this file to freeze what LLP-030 is actually doing in Stage 2:
turning the existing strategy, business-plan, and financial-model work into a
bounded, governed strategic control packet that ML1 can use to make firm-level
decisions.

The root files remain substantive governed work product. The planning packet
does not replace them. It makes their assumptions, dependencies, risks,
communication loops, and gate criteria explicit.

## In Scope

- normalize LLP-030 into a canonical planning-stage packet
- lock the intended role of `FIRM_STRATEGY.md`, `BUSINESS_PLAN.md`, and
  `FINANCIAL_MODEL.md`
- freeze the current operating baseline of `CAD 240,000` revenue and
  `CAD 80,000` owner compensation as the working planning anchor unless ML1
  changes it
- define the channel-sequencing logic for `F01 -> F02 -> F03`
- define the intended offer stack and target-client posture for `ICP-01` and
  `ICP-02`
- define the staffing-sequence logic: setter first if needed for intake,
  associate-lawyer delivery layer before any senior-lawyer layer
- define the profitability and margin questions that the financial model must
  answer before leverage is activated
- define the cross-project alignment boundary with `LLP-002`, `LLP-011`,
  `LLP-012`, `LLP-025`, and `LLP-033`
- define the cross-entity treatment of Andersen work in the Levine Law
  operating picture
- define the planning-stage metrics and ML1 gate criteria for this project

## Out of Scope

- execution authority for funnel build, spend, hiring, or delivery changes
- changing firm strategy by implication through downstream projects
- approving live marketing, staffing, or pricing decisions without explicit
  ML1 sign-off
- matter-level intake or delivery operations
- replacing `LLP-002` as the source of budget fact
- replacing `17513721 Canada Inc` entity treatment with LL-only assumptions
- treating the current `FINANCIAL_MODEL.md` stub as an approved or complete
  decision model

## Gate Criteria For Planning Completion

Planning is complete only when:

- the canonical Stage 2 packet exists and is internally consistent
- `FIRM_STRATEGY.md`, `BUSINESS_PLAN.md`, and `FINANCIAL_MODEL.md` are aligned
  on target state, staffing sequence, and profitability logic
- the dependent-project links are explicit and current
- the Andersen cross-entity treatment is documented clearly enough to avoid
  omission, misclassification, or double counting
- the financial-model open items are narrow enough for ML1 to use the project
  as a real decision layer rather than a narrative placeholder
- executing remains unauthorized unless ML1 separately closes the
  `Planning -> Executing` gate
